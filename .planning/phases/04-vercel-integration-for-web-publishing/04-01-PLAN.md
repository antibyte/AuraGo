  <action>
Add a new `vercel` config block that mirrors the Netlify integration pattern while using Vercel-specific field names. Include at minimum:
- `enabled`
- `readonly`
- `allow_deploy`
- `allow_project_management`
- `allow_env_management`
- `allow_domain_management`
- `default_project_id`
- `team_id`
- `team_slug`

Wire `vercel.token` to the vault key `vercel_token` in config handlers and secret extraction. Load the vault value into the runtime config during startup. Add the Vercel token to the Python secret denylist so it cannot be exported to Python tools. Ensure config defaults do not enable the integration or mutation permissions by default unless the existing SaaS provider pattern explicitly does so.
  </action>
  <acceptance_criteria>
- `config_template.yaml` contains a `vercel:` block with all listed fields
- `internal/config/config_types.go` defines a Vercel config struct/section containing `AllowProjectManagement`, `AllowEnvManagement`, and `AllowDomainManagement`
- `internal/server/config_handlers_main.go` maps `vercel.token` to `vercel_token`
- `internal/tools/python_secrets.go` contains `vercel_token`
- `internal/config/config_migrate.go` or equivalent vault load path reads `vercel_token`
  </acceptance_criteria>
</task>

Create Vercel server handlers equivalent to the Netlify status/test flow. Add:
- `GET /api/vercel/status`

The test endpoint should validate token reachability against Vercel and enrich the response with lightweight account/project information when possible. Register both routes in the config routes file.

- `internal/server/server_routes_config.go` contains `/api/vercel/test-connection`
