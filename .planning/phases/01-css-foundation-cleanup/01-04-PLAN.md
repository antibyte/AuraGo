<task type="auto">
  <name>Task 2: Document override rules in shared.css</name>
  <files>ui/shared.css</files>
  <action>
Add a "Specificity and Override Rules" section to the shared.css header (after the naming convention section added in Plan 03).

Document:
1. Base specificity: shared.css uses single-class selectors (.card, .badge, .btn) -- these have low specificity
2. Page-specific overrides: When a page needs to change a shared component, use ONE of these approaches:
   a) Prefer: Add a page-specific modifier class on the container (e.g., .missions-page .card {}) -- this wins via cascade without conflict
   b) Avoid: Directly redefining .card in the page CSS -- this creates duplication and specificity wars
   c) New components: Use .aura-* prefix per naming convention (D-01/D-04)
3. The golden rule: shared.css defines BASE styles; page CSS defines CONTEXT styles via descendant selectors
4. Bad patterns to avoid: redefining base component classes in page CSS

Also add a section at the start of the components section in shared.css explaining:
- These are base/atomic components
- They use single-class selectors for easy overriding
- Page-specific styles should use .page-name .component pattern
  </action>
  <verify>
    <automated>grep -c "Override Rules\|Specificity" ui/shared.css</automated>
  </verify>
  <done>shared.css has a documented "Override Rules" section in its header</done>
</task>
