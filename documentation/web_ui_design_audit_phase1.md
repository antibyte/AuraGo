# Web UI Design Audit – Phase 1

## Objective
Establish a shared, enforceable design baseline for the AuraGo Web UI before implementation refactors.

Phase 1 produces:
1. A UI inventory (pages + repeated components)
2. A design token policy
3. A practical Do/Don't guide for contributors
4. A scoring checklist for consistency audits
5. A prioritized backlog for Phase 2 execution

## 2) Design Token Policy (Baseline)

### 2.1 Token ownership
Use `ui/shared.css` as single source of truth for visual tokens:
- Color (`--accent`, `--text-primary`, etc.)
- Surface/background (`--bg-primary`, `--bg-secondary`, etc.)
- Borders and shadows
- Radius and typography scales

### 2.2 Token usage rules
- No hardcoded colors in markup except runtime data visualizations where unavoidable.
- No inline spacing values (`margin`, `padding`, `gap`) in HTML.
- New components must consume existing tokens first; add new tokens only when needed by at least 2 contexts.

### 2.3 State naming conventions
Use unified class names for visibility/state:
- `.is-hidden`
- `.is-loading`
- `.is-active`
- `.is-disabled`

Avoid ad-hoc inline state toggles unless generated by JS for dynamic values.

---

## 3) Contributor Do/Don't Guide

### ✅ Do
- Reuse shared component classes before adding page-local styles.
- Keep brand/header structure identical across pages.
- Implement dark/light support through tokens, not duplicated rules.
- Add semantic classes for variants (`btn--danger`, `card--compact`, etc.).
- Keep copy and labels consistent across screens.

### ❌ Don’t
- Don’t add `style="..."` for static styling in HTML.
- Don’t introduce one-off button/input styles when a shared variant can cover it.
- Don’t mix unrelated spacing systems (Tailwind utility + local inline + custom one-off class) in one component.
- Don’t introduce new visual color values without token review.

## 4) Audit Scorecard (0–5 per criterion)

Use this checklist per page:

1. **Brand consistency**
   - Logo mark, title casing, subtitle style, header spacing.
2. **Token compliance**
   - No hardcoded visual values where tokens exist.
3. **Component reuse**
   - Buttons/forms/cards/states use shared classes.
4. **Layout rhythm**
   - Consistent spacing scale and section cadence.
5. **Interaction consistency**
   - Hover/focus/active/disabled feel aligned.
6. **State clarity**
   - Loading/empty/error/success states are coherent.
7. **Theme parity**
   - Dark and light themes both readable and balanced.
8. **Accessibility baseline**
   - Keyboard focus visibility, contrast, semantic labels.

### Rating interpretation
- **0–1**: major inconsistency, high redesign need
- **2–3**: partially aligned, medium refactor required
- **4**: mostly consistent, minor cleanup
- **5**: fully aligned with system

## 6) Execution workflow for design reviews

1. Pick one page
2. Score with checklist (8 criteria)
3. Capture top 3 inconsistencies
4. Map to shared component/token fix
5. Implement and re-score
6. Log before/after score in PR description

---

## 7) Definition of Done for Phase 1

Phase 1 is complete when:
- Inventory and standards are documented
- Scorecard is agreed and usable by contributors
- A concrete P0/P1/P2 backlog exists
- Team can start Phase 2 without redefining design principles

### Key observations from baseline
1. Header branding is mostly recognizable, but subtitle style and spacing differ by page.
2. Inline styles are the main source of token non-compliance.
3. Visibility/state handling is often done with inline `display:none` instead of shared state classes.
4. Form spacing and helper text rhythm differ significantly between setup/config/missions/invasion.
